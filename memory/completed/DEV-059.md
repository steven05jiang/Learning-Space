## Requirements

Add a "Re-process" button to the resource detail page that lets users manually trigger reprocessing of a resource. This is the only way to retry a failed resource or force a re-summarization. The button resets `processing_status` to `pending` and enqueues a new worker job.

### Acceptance Criteria

- [x] `POST /resources/{id}/reprocess` endpoint added; resets `processing_status = pending` and enqueues worker job; returns 202
- [x] Endpoint returns 401 if unauthenticated, 403 if resource belongs to another user
- [x] "Re-process" button visible on resource detail page
- [x] Button is disabled (grayed out) while `processing_status = processing`
- [x] Button click calls POST /resources/{id}/reprocess; shows toast confirmation on 202
- [x] Resource detail page refreshes `processing_status` after re-process is triggered
- [x] Unit test for new endpoint (success, 401, 403)

## Implementation Summary

Added POST /resources/{id}/reprocess endpoint with proper 403 (not 404) for wrong-owner access. Frontend Re-process button on resource detail page with disabled state during processing and loading feedback. Fixed round-1 issues: 403 vs 404 ownership check, mock enqueue assertion in tests, removed build artifacts from git. All 342 unit tests pass.
