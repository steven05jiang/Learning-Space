## Requirements

Add the `GET /api/v1/resources/search` FastAPI endpoint. Thin wrapper around `ResourceSearchService`. Validates query parameters (`q`, `tag`, `limit`, `offset`) via Pydantic, resolves `owner_id` from session, calls the service, and serializes the result as `ResourceSearchResponse`. Includes `rank` field in each resource object.

### Acceptance Criteria

- [x] `GET /api/v1/resources/search?q=...` returns 200 + `{resources: [...], total: N}`
- [x] `ResourceSearchRequest` Pydantic schema: `q` (required, 1–500 chars, strip_whitespace), `tag` (optional), `limit` (default 20, 1–100), `offset` (default 0)
- [x] `ResourceSearchResponse` Pydantic schema: `resources: List[ResourceWithRank]`, `total: int`
- [x] Empty `q` → 400 `SEARCH_QUERY_EMPTY`
- [x] `q` > 500 chars → 400 `SEARCH_QUERY_TOO_LONG`
- [x] Unauthenticated → 401
- [x] `limit` / `offset` out of range → 422 (FastAPI default)
- [x] Endpoint registered in router under `/resources` prefix
- [x] `rank` float field included in each resource in response

## Implementation Summary

Added GET /resources/search endpoint as a thin wrapper around ResourceSearchService. Introduced ResourceWithRank (extends ResourceResponse + rank field), ResourceSearchRequest, and ResourceSearchResponse Pydantic schemas. Custom 400 error codes for business logic validation (SEARCH_QUERY_EMPTY, SEARCH_QUERY_TOO_LONG). 9 test cases covering all paths. All CI checks passed.
