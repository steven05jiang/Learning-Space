## Requirements

When a resource is deleted, the knowledge graph must be updated to decrement/remove the tag relationships that were backed by that resource. This ensures graph consistency when users delete resources.

### What to implement

1. **Update the `sync_graph` stub in `apps/api/workers/tasks.py`** — replace the stub with real logic that calls `graph_service.remove_resource_tags(owner_id, tags)` and `graph_service.cleanup_orphan_tags(owner_id)` for the "delete" operation.

2. **Hook deletion into the resource DELETE endpoint** — in `apps/api/routers/resources.py`, after a resource is deleted from PostgreSQL, enqueue a `sync_graph` job (with `operation="delete"`, passing the resource's `owner_id` and `tags`) via the queue service.

3. **Unit tests** — add tests for:
   - `sync_graph` with `operation="delete"` calls `graph_service.remove_resource_tags` and `cleanup_orphan_tags`
   - DELETE /resources/{id} enqueues a graph sync job after deletion

## Implementation Summary

Updated `sync_graph` task in `tasks.py` to handle delete operations — calls `graph_service.remove_resource_tags(owner_id, tags)` followed by `cleanup_orphan_tags(owner_id)` in a non-blocking try/except. Updated DELETE endpoint in `resources.py` to capture resource tags before deletion and enqueue a `sync_graph` job after successful PostgreSQL deletion. Enhanced `queue_service` to support additional parameters for task payloads. Added 7 unit tests for `sync_graph` and 3 tests for the DELETE endpoint. All acceptance criteria met, lint and tests passing.
