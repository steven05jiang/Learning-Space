## Requirements

Gate access at the OAuth callback: if the authenticated user's email is not in the
`ALLOWED_EMAILS` allowlist, redirect them to `/coming-soon` instead of completing login.

### Config change: `apps/api/core/config.py`

Add a new optional setting:
```python
# Comma-separated list of allowed emails. Empty = allow all (open access).
allowed_emails: str = ""
```

Add a helper method or property to parse it:
```python
@property
def allowed_emails_set(self) -> set[str]:
    if not self.allowed_emails:
        return set()
    return {e.strip().lower() for e in self.allowed_emails.split(",") if e.strip()}
```

### Auth router change: `apps/api/routers/auth.py`

In the `oauth_callback` handler (GET /callback/{provider}):
1. After successfully authenticating the user and getting their email
2. If `settings.allowed_emails_set` is non-empty AND user email (lowercased) is NOT in the set:
   - Return a redirect response to `{frontend_base_url}/coming-soon`
   - Do NOT set the session cookie / JWT
   - Log a warning: `logger.warning("Blocked non-allowlisted login attempt: %s", email)`

The redirect URL should use the same `_get_frontend_base_url` / `oauth_redirect_base_url` logic
already in the router to determine the correct frontend host.

### Env var
- `ALLOWED_EMAILS=user1@gmail.com,user2@gmail.com`
- Empty or absent = open access (no restriction) — safe default for local dev

### Unit test (optional but preferred)
- Add a test in `apps/api/tests/` verifying that an unlisted email triggers redirect to /coming-soon
- Use the existing test fixtures/patterns

### Acceptance criteria
- `ALLOWED_EMAILS` is parsed correctly (comma-separated, trimmed, lowercased)
- Non-allowlisted email → redirect to /coming-soon (no session created)
- Allowlisted email → normal login flow
- Empty `ALLOWED_EMAILS` → no restriction (all users allowed)
- `make api-lint` passes
- `make api-test` passes
