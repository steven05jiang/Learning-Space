## Requirements

**Type:** Backend
**BDD Reference:** All worker scenarios (INT-024–028)
**Description:** Orchestrate the full processing pipeline as an async worker job:
1. Set status → PROCESSING
2. Fetch content — unauthenticated URL fetcher (DEV-020) for URL resources; `original_content` for text resources. DEV-021 deferred.
3. LLM processing (DEV-022) — extract title, summary, tags
4. Update resource in DB (status → READY, write title/summary/tags)
5. Trigger graph update (hook present; DEV-026 will wire it fully)
6. Handle errors at each step → status → FAILED with status_message

**Acceptance Criteria:**
- [x] Pipeline runs end-to-end for URL resources (unauthenticated fetch only)
- [x] Pipeline runs end-to-end for text resources (skip fetch step, use original_content)
- [x] Status transitions: PENDING → PROCESSING → READY or FAILED
- [x] LLM failure sets status=FAILED with descriptive status_message
- [x] Fetch failure sets status=FAILED with descriptive status_message
- [x] Graph update hook is present (≥2 tag check, graceful failure handling)

**Dependencies:** DEV-019 ✅, DEV-020 ✅, DEV-022 ✅, DEV-025 ✅

## Implementation Summary

6-stage pipeline in `apps/api/workers/tasks.py`. Added `status_message` field to Resource model with Alembic migration. Uses naive datetimes (`datetime.utcnow()`) throughout to match DB schema. Graph update hook calls GraphService (DEV-025) when ≥2 tags are available; failures are logged but don't block pipeline. 9 unit tests in `apps/api/tests/test_process_resource.py`; all 222 unit tests + 33 integration tests pass.
