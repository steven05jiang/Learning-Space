## Requirements

Replace the flat `Tag` Neo4j schema with a three-level node hierarchy:
- **Root** nodes — system/user root categories (mirror the `categories` PostgreSQL table)
- **Category** nodes — LLM-generated mid-level topic groupings
- **Tag** nodes — specific topic tags (leaf level)

Add `CHILD_OF` edges (Tag → Category, Category → Root) and `BELONGS_TO` edges (Resource → Root, per `top_level_categories`). Update the graph service and worker pipeline to create/merge all three node types correctly. Update the GET /graph response to include `node_type`.

Also update INT-029–035 integration tests which assert on the old flat Tag schema.

Design spec: `docs/design-category-taxonomy.md`

### Acceptance Criteria

- [x] `Root`, `Category`, `Tag` node types exist in Neo4j; each has a `node_type` property (`root` | `category` | `topic`)
- [x] `CHILD_OF` relationship: Tag → Category, Category → Root
- [x] `BELONGS_TO` relationship: Resource → Root (one per entry in `top_level_categories`)
- [x] `graph_service.update_graph()` creates/merges Root and Category nodes in addition to Tag nodes
- [x] `GET /graph` response includes `node_type` field on each node
- [x] Worker pipeline updated to pass `top_level_categories` to graph service
- [x] INT-029–035 integration tests updated to pass with new schema
- [x] Unit tests for graph service covering all three node levels

## Implementation Summary

Implemented three-level Neo4j hierarchy (Root → Category → Tag) with CHILD_OF and BELONGS_TO relationships. Added `update_graph()` to GraphService, updated worker to pass `top_level_categories` from LLM, added `top_level_categories` to LLMResult dataclass and resource model. Fixed owner_id type consistency (always `str()`) across all graph service methods. Updated INT-029–035 and unit tests.
