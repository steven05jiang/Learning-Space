## Requirements

Update the LLM processing prompt to:
1. Include the user's existing tag list as context so the model can reuse tags instead of generating new variants
2. Include the valid root category names (from the `categories` table) so the model can assign them
3. Require a `top_level_categories` field in the LLM JSON output (non-empty list of valid root category names)

Add validation of LLM output against the categories table; raise structured errors on invalid or missing categories.

Design spec: `docs/design-category-taxonomy.md`

### Acceptance Criteria

- [x] LLM prompt includes the user's existing tag list (fetched from Neo4j before LLM call)
- [x] LLM prompt includes the list of valid root category names (fetched from `categories` table)
- [x] LLM JSON output schema requires `top_level_categories` (list of strings, at least one entry)
- [x] Post-LLM validation: if any value in `top_level_categories` is not a valid category name → raise `INVALID_CATEGORY` error, resource marked `failed`
- [x] Post-LLM validation: if `top_level_categories` is empty or missing → raise `CATEGORY_REQUIRED` error, resource marked `failed`
- [x] `resources.top_level_categories` JSONB field stored after successful LLM call
- [x] Unit tests: valid output, INVALID_CATEGORY, CATEGORY_REQUIRED

## Implementation Summary

Added `get_user_tags()` to GraphService to fetch existing user tags from Neo4j. Updated worker to fetch user tags and valid categories before LLM call. Updated LLM processor to include both in the system prompt and validate category assignments (INVALID_CATEGORY, CATEGORY_REQUIRED errors). Added 8 dedicated category unit tests and updated 25+ existing tests to mock new pre-LLM calls.
