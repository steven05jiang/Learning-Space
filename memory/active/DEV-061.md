## Requirements

Replace the flat `Tag` Neo4j schema with a three-level node hierarchy:
- **Root** nodes — system/user root categories (mirror the `categories` PostgreSQL table)
- **Category** nodes — LLM-generated mid-level topic groupings
- **Tag** nodes — specific topic tags (leaf level)

Add `CHILD_OF` edges (Tag → Category, Category → Root) and `BELONGS_TO` edges (Resource → Root, per `top_level_categories`). Update the graph service and worker pipeline to create/merge all three node types correctly. Update the GET /graph response to include `node_type`.

Also update INT-029–035 integration tests which assert on the old flat Tag schema.

Design spec: `docs/design-category-taxonomy.md`

### Acceptance Criteria

- [ ] `Root`, `Category`, `Tag` node types exist in Neo4j; each has a `node_type` property (`root` | `category` | `topic`)
- [ ] `CHILD_OF` relationship: Tag → Category, Category → Root
- [ ] `BELONGS_TO` relationship: Resource → Root (one per entry in `top_level_categories`)
- [ ] `graph_service.update_graph()` creates/merges Root and Category nodes in addition to Tag nodes
- [ ] `GET /graph` response includes `node_type` field on each node
- [ ] Worker pipeline updated to pass `top_level_categories` to graph service
- [ ] INT-029–035 integration tests updated to pass with new schema
- [ ] Unit tests for graph service covering all three node levels
