# BUG-008: Google OAuth login does not create a local user account in the database

**Status:** 🔄 Active
**Priority:** P0 — Critical
**Reported:** 2026-03-17
**Branch:** (pending)
**PR:** (pending)

## Description

When a user logs in via Google OAuth, no local user record is created in the PostgreSQL `users` table (or the `accounts` table is not populated with the provider association). As a result:
- The same identity cannot be linked to other social providers later
- Account management features (link/unlink) have no local user to anchor to
- BUG-007 (linked accounts showing wrong status) is likely a downstream symptom

## Expected Behavior

On first Google OAuth login (and any subsequent OAuth login):
1. If no local user exists for this Google email, create one in the `users` table
2. Create/upsert a record in the `accounts` (or `oauth_accounts`) table linking the local user → Google provider + provider user ID + access token
3. Return a JWT tied to the local user ID (not the Google sub directly)

This design allows the same local user to later link additional providers (GitHub, Apple, etc.) via the account linking flow (DEV-007/DEV-008).

## Acceptance Criteria

- [ ] First-time Google login creates a row in `users` table
- [ ] A corresponding row in `accounts` (or equivalent) links `user_id → provider=google, provider_user_id=<google_sub>`
- [ ] Subsequent Google logins find the existing user (no duplicate creation)
- [ ] JWT issued is tied to the local `user_id`
- [ ] Account linking (DEV-007) can now successfully associate a second provider to the same local user
- [ ] Settings page linked accounts reflects the correct state (unblocks BUG-007)
