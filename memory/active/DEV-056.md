## Requirements

Replace the single HTTP fetch in the worker pipeline with a tiered strategy:
1. **Domain blocklist** — skip fetch entirely for known-blocked domains (configurable YAML)
2. **Tier 1 — API fetch** — use official platform API where available (YouTube, Twitter/X)
3. **Tier 2a — HTTP fetch** — direct HTTP request (current default fast path)
4. **Tier 2b — Playwright fetch** — headless browser fallback for bot-blocked pages

Try each tier in order and stop at the first success. Record which tier succeeded and any error type.

## Acceptance Criteria

- [x] Domain blocklist config (YAML) loaded at startup; blocked domains skip fetch immediately
- [~] Tier 1 API fetch invoked for known platform domains (YouTube, Twitter/X) when user has linked account (stub only; no integration yet)
- [x] Tier 2a HTTP fetch attempted for all other URLs
- [x] Tier 2b Playwright headless fetch invoked when HTTP returns 403 or other bot-blocked indicator
- [x] `fetch_tier` field (values: `api` | `http` | `playwright`) stored on resource after fetch completes
- [x] `fetch_error_type` field stored on resource when fetch fails (e.g. `BOT_BLOCKED`, `AUTH_REQUIRED`, `FETCH_ERROR`)
- [x] Alembic migration adds `fetch_tier` and `fetch_error_type` columns to `resources` table
- [x] Worker logs which tier succeeded (or failed) per resource
- [ ] Playwright available in worker Docker image (playwright + chromium) — NOT INCLUDED IN THIS PR
- [~] Unit tests cover: blocklist hit, HTTP success, HTTP blocked → Playwright fallback, all tiers fail (unit tests present; missing integration test)
