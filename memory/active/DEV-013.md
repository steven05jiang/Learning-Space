## Requirements

Implement `POST /resources` endpoint that:
- Requires authentication via `get_current_user` dependency (DEV-006) — return 401 if not authenticated
- Accepts request body with `content_type` (`url` or `text`), `original_content` (required), and `prefer_provider` (optional)
- Creates a resource row in PostgreSQL with: correct `owner_id` (from current_user), `content_type`, `original_content`, `prefer_provider`, `status=PENDING`
- Enqueues a background processing job for the resource
- Returns HTTP 202 with the full resource response (matching `ResourceResponse` Pydantic model from DEV-012)

**BDD Scenarios covered:**
- Authenticated user submits a URL resource → 202, status PENDING/PROCESSING, job enqueued
- Authenticated user submits a text resource → 202, status PENDING/PROCESSING
- Unauthenticated user attempts to create a resource → 401
- User submits a URL with prefer_provider hint → resource stored with prefer_provider for worker

**Acceptance Criteria:**
- [ ] Resource row created with correct `owner_id`, `content_type`, `original_content`, `prefer_provider`
- [ ] Status is PENDING or PROCESSING
- [ ] Background job is enqueued
- [ ] Returns 202 with full resource response
- [ ] Returns 401 if not authenticated

**Depends on:** DEV-006 (merged PR #16), DEV-012 (merged PR #10), DEV-002 (merged PR #7)

**Note on task queue:** DEV-019 (task queue infrastructure) is not yet implemented. For now, enqueue the job using FastAPI `BackgroundTasks` (stdlib async background task) as a placeholder — this keeps the endpoint functional and testable until the full queue (Celery/ARQ) is built in DEV-019. Log the job dispatch.
