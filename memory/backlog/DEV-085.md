## Description

Implement the two Discover API endpoints. `GET /discover/bookmarks` returns paginated `twitter_bookmarks` rows where `is_added=false`, joined with `twitter_posts`, sorted by `bookmarked_at DESC`. `POST /discover/bookmarks/{tweet_id}/add` delegates to the internal `POST /resources` creation flow (with `content_type=url`, `prefer_provider=twitter`), sets `is_added=true` on success, and triggers an async post-cleanup check for this `tweet_id`.

## Acceptance Criteria

- [ ] `GET /discover/bookmarks` returns `{ total, limit, offset, items[] }`
- [ ] Each item includes: `tweet_id`, `tweet_url`, `author_name`, `author_username`, `title` (null for tweets), `preview_text`, `posted_at`, `bookmarked_at`
- [ ] Only `is_added=false` rows returned; sorted by `bookmarked_at DESC`
- [ ] `limit` (default 20) and `offset` (default 0) query params respected
- [ ] `POST /discover/bookmarks/{tweet_id}/add` calls internal resource creation with `content_type=url`, `original_content=tweet_url`, `prefer_provider=twitter`
- [ ] On resource creation success: `is_added=true` set; 202 returned with resource object
- [ ] Idempotent: returns 200 if `is_added` already true
- [ ] Returns 404 if `tweet_id` not in current user's `twitter_bookmarks`
- [ ] Returns 422 if resource creation fails (body includes user-facing `detail`)
- [ ] Both endpoints require authentication (401 if unauthenticated)
- [ ] Async cleanup for `tweet_id` triggered after response (best-effort, non-blocking)
