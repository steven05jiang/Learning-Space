## Description

Implement the Twitter API call for fetching tweet/Article content as part of the Tier 1 fetch path in the worker pipeline. When a resource URL is from `twitter.com` or `x.com`, the worker calls `GET /2/tweets/{id}` (with Article fields for long-form content) using the user's linked `access_token`. Handles token refresh on 401. Integrates into the existing tiered fetch strategy. Supersedes DEV-021.

## Acceptance Criteria

- [ ] Tier 1 fetch for `twitter.com`/`x.com` URLs calls Twitter API (`GET /2/tweets/{id}?tweet.fields=id,text,created_at,author_id&expansions=author_id&article.fields=title,body`)
- [ ] Uses `user_accounts.access_token`; attempts token refresh via `refresh_token` on 401
- [ ] If refresh fails: sets `fetch_error_type=API_REQUIRED`, `status_message` prompts user to re-authorize in Settings
- [ ] Sets `resource.fetch_tier=api` on success
- [ ] For Articles: extracts `article.title` and `article.body` as content for LLM processing
- [ ] For standard tweets: uses `text` field as content
- [ ] DEV-021 marked as superseded in dev-tracker (comment added)
